Mấy tuần qua mình đụng phải một trong những quyết định khó nhất khi build sản phẩm một mình: port toàn bộ VividKit Desktop — từ stack Tauri v2 + Rust sang Wails v3 + Go. § 01

Mode vs Flag — hai khái niệm khác nhau

Trước khi đi vào từng tính năng, cần phân biệt một điểm dễ nhầm. /ck:plan nhận hai loại tham số hoàn toàn khác nhau về cách chúng tương tác: mode (chọn pipeline) và composable flag (add-on ghép vào mode). Nhầm lẫn hai cái này dẫn đến những lệnh không hợp lệ hoặc kết quả không như kỳ vọng.

Cụ thể: --deep là mode, còn --tdd là flag. Điều này có nghĩa --deep --tdd hoàn toàn hợp lệ (mình dùng combo này), nhưng --deep --hard thì không — hai mode loại trừ nhau.

Loại Ví dụ Tác dụng
Mode --fast, --hard, --deep, --parallel, --two Chọn toàn bộ pipeline. Chỉ 1 mode per lệnh, mutual exclusive.
Composable flag --tdd, --no-tasks Add-on. Ghép được với bất kỳ mode, không thay đổi pipeline — chỉ thay đổi nội dung output.
argument-hint ck-plan/SKILL.md 
# Format chính thức từ skill definition
argument-hint: "[task] [--fast|--hard|--deep|--parallel|--two] [--tdd|--no-tasks]"

# Dấu | giữa modes = exclusive. --tdd đứng ở block riêng = composable.
# Combo hợp lệ:
/ck:plan "port vividkit" --deep --tdd

# Combo KHÔNG hợp lệ:
/ck:plan "port vividkit" --deep --hard  # ← 2 modes cùng lúc
§ 02

--deep — khi scope là "đại tu" chứ không phải "sửa"

--deep không phải là --hard "xịn hơn". Đây là mode được thiết kế riêng cho một tình huống cụ thể: task chạm vào 5+ vùng code độc lập, có quyết định kiến trúc mới phải đưa ra ở mỗi phase, và sợ rằng một cái nhìn tổng lúc đầu sẽ không đủ chi tiết khi cần implementation.

Điểm khác biệt cốt lõi so với --hard: thay vì scout một lần cho toàn bộ codebase, --deep chạy thêm một vòng scout riêng cho từng phase. Đắt về token, nhưng output là phase file đủ chi tiết để cook không phải tự đoán.

Pipeline của --deep

  1. Spawn 2–3 researcher agents cho high-level architecture analysis song song
  2. Đọc docs liên quan + chạy /ck:scout qua tất cả affected areas
  3. Per-phase scout — điểm cốt lõi: với mỗi phase đã plan, chạy focused scout: inventory files create/modify/delete, count existing tests, list functions cần test protection
  4. Planner embed toàn bộ scout data trực tiếp vào từng phase file
  5. Red-team review để bắt missing risk hoặc assumption sai
  6. Post-plan validation chéo giữa phases
  7. Hydrate tasks (trừ khi bật --no-tasks)
  8. Output lệnh cook: /ck:cook {path}/plan.md

--deep không giúp code nhanh hơn. Nó giúp cook biết chính xác file nào đụng, pattern nào dùng, parallel group nào chạy được song song. Với port đa-stack như Rust → Go, đây là ROI thật.

— Từ notes viết lại sau khi port xong Phase 4
Điều kiện--hard--deep
Touch 1–4 vùng code Đủ dùng Overkill
Touch 5+ vùng, kiến trúc mới Thiếu depth Đúng lựa chọn
Codebase đang có user data Rủi ro bỏ sót Per-phase scout bắt được
§ 03

--tdd — khi sợ refactor phá hành vi đang chạy

Default flow của /ck:cook: AI đọc plan, execute phase tasks, viết test sau khi code đã sửa. Với greenfield — hoàn toàn ổn. Với refactor hoặc port từ stack khác — nguy hiểm. Behavior đang tồn tại trong code cũ có thể bị thay đổi vô tình, và test viết sau không bắt được điều này vì test sẽ bám theo behavior mới.

--tdd đảo ngược thứ tự: phase file được tạo ra với 4 section bổ sung, trong đó Tests Before phải viết xong và pass trước khi bất kỳ dòng refactor nào được gõ. Tests đó document behavior hiện tại — khi refactor xong, chúng trở thành regression gate.

SectionVai trò trong phase file
Tests Before Regression coverage viết trước khi refactor. Document hành vi hiện tại bằng cách assert output cụ thể.
Refactor Mô tả code changes được bảo vệ bởi Tests Before. Cook không được làm Tests Before fail.
Tests After Test cho hành vi mới được thêm vào trong phase này (nếu có). Optional.
Regression Gate Lệnh compile + test cụ thể phải pass sau refactor. Hard requirement — phase không hoàn thành nếu gate fail.
bash Lệnh mình đã gõ 
# Port Wails — plan + tdd cho phases có Rust equivalent
/ck:plan --deep --tdd plans/260412-0239-port-vividkit-wails3/plan.md

# Refactor UI đã live — regression gate bắt buộc
/ck:cook plans/260414-1247-brainstorm-ui-redesign/ --tdd
§ 04

Một lưu ý — khi nào dùng /ck:brainstorm trước

--deep giỏi trong việc lập kế hoạch chi tiết khi bạn đã biết mình muốn gì. Nhưng nếu bạn chưa chắc về hướng đi — kiến trúc nào, trade-off nào, thư viện nào — thì --deep sẽ tạo ra một plan rất chi tiết cho một quyết định chưa đúng. Trong trường hợp đó, /ck:brainstorm trước là lựa chọn tốt hơn.

Tình huốngLuồng nên dùngLý do
Biết rõ muốn gì, cần plan chi tiết --deep trực tiếp Plan sẽ accurate ngay từ đầu
Chưa chắc về kiến trúc / trade-off /ck:brainstorm → --deep Brainstorm làm rõ hướng trước khi plan đốt token
Cần compare 2 approach cụ thể /ck:brainstorm → --two → --deep --two tạo 2 plan song song để so sánh, rồi chọn cái đi sâu
§ 05

Decision matrix — chọn mode và flag nào

Ba dòng cuối (★) là combo recommendations — dùng khi cần kết hợp mode + flag.

Tình huống Lệnh Token cost
Task nhỏ, scope rõ ràng, 1–2 file --fast Thấp
Feature trung bình, 3–4 area --hard Trung bình
Greenfield lớn, nhiều unknowns --hard Trung bình
Refactor nhỏ, behavior quan trọng --fast --tdd Thấp
Refactor feature đang live, user đã dùng --hard --tdd Trung bình
Major refactor, 5+ area, architectural debt --deep Cao
Port đa-stack, không biết Go idiom map thế nào --deep Cao
★ Port đa-stack + giữ behavior cũ --deep --tdd Cao nhất
★ Parallel team, risk ownership conflict --deep --parallel --tdd Rất cao
★ Chưa chắc hướng → explore trước /ck:brainstorm → --deep --tdd Cao (2 bước)
§ 06

Inside — cơ chế bên trong

Phần này đào sâu vào cách --deep và --tdd thực sự được implement trong skill. Không bắt buộc phải đọc để dùng được, nhưng giúp hiểu tại sao output lại có cấu trúc như vậy và tại sao token cost cao đến thế.

6.1

Mode activation table

Mỗi mode kích hoạt một tập agent và sub-process khác nhau. Bảng dưới so sánh các thành phần được spawn theo mode — đây là lý do chi phí token khác nhau đáng kể.

Mode Researcher Red Team Validation Per-phase scout
--fast – – – –
--hard 1 – Post-plan –
--deep 2–3 ✓ Post-plan ✓ per phase
--parallel 2–3 ✓ Cross-team –
--two 1 per plan Compare Diff output –
6.2

Per-phase scout — scout data embed vào phase file

Với mỗi phase đã draft, --deep spawn một scout agent chạy /ck:scout focused vào đúng files/dirs thuộc phase đó. Scout output được embed trực tiếp vào phase file dưới dạng structured sections: file inventory table, existing test count, function/interface checklist cần test bảo vệ. Cook đọc những thông tin này thay vì phải tự suy luận từ codebase.

6.3

--tdd injection vào phase template

Khi flag --tdd được bật, planner không chỉ thêm 4 section vào phase file. Nó còn thay đổi thứ tự của cook's execution plan: cook/references/workflow-steps.md Step 3 được replace bởi một "TDD loop" — viết Tests Before → run → assert pass → refactor → assert Tests Before vẫn pass → viết Tests After.

6.4

Tại sao --deep không có trong --fast hoặc --hard

Per-phase scout về cơ bản là một nested planning loop: plan toàn cục tạo ra N phases, rồi N sub-plan được tạo ra cho từng phase. Mỗi sub-plan cần scout context riêng. Chi phí là O(N) lần scout thay vì O(1). Với task nhỏ (1–3 phase), overhead này không đáng. Với 9-phase port như mình, nó là investment đáng.

§ 07

Cách chạy và vài mẫu prompt

Ba pattern mình dùng nhiều nhất trong thực tế — từ simple đến phức tạp. Tất cả đều test được ngay trong terminal.

bash Pattern 1 — Fresh plan với --deep --tdd 
/ck:plan "port vividkit desktop từ Tauri v2 + Rust sang Wails v3 + Go,
giữ nguyên React frontend, chỉ rewrite backend và IPC adapter.
13 module, có PTY, NDJSON streaming, SQLite, git worktree." --deep --tdd
bash Pattern 2 — Refine plan đã có 
# Nếu đã có plan.md từ --hard nhưng muốn deep-dive thêm
/ck:plan --deep --tdd plans/260412-0239-port-vividkit-wails3/plan.md
bash Pattern 3 — Cook với --tdd override 
# Plan không có --tdd nhưng muốn cook dùng TDD flow
# --tdd ở cook level override plan-level setting
/ck:cook plans/260414-1247-brainstorm-ui-redesign/ --tdd
  • Task description càng cụ thể, researcher agents càng ít phải đoán context
  • Mention explicitly các constraint quan trọng: "backward compat với user data cũ", "giữ API contract"
  • Nếu dùng --deep lần đầu cho project, expect 10–20 phút wait time cho per-phase scout
  • Phase file output sẽ có file inventory table — review trước khi chạy cook để đảm bảo scope đúng
§ 08

Best practices

  1. 1

    Đặt task description trước mode

    Thứ tự argument matter. /ck:plan "task mô tả" --deep --tdd cho kết quả tốt hơn /ck:plan --deep --tdd "task mô tả" vì parser của skill đọc theo thứ tự. Task description là input chính — đừng hy sinh nó vì ngại gõ dài.

  2. 2

    Review file inventory trước khi cook

    --deep tạo ra file inventory table trong mỗi phase file. Đọc qua cột "action" (create/modify/delete) và verify không có file nào bị scope sai. Sửa sai ở plan dễ hơn rollback sau khi cook đã chạy xong.

  3. 3

    Không bật --tdd cho greenfield module

    --tdd designed cho "bảo vệ behavior hiện tại". Với code mới 100%, Tests Before section sẽ trống hoặc trivial — cook mất thời gian viết test cho code chưa tồn tại. Dùng --hard cho greenfield, reserve --tdd cho phases có code cũ cần giữ behavior.

  4. 4

    Kiểm tra regression gate sau mỗi phase

    Mỗi phase file có Regression Gate command cụ thể. Chạy nó ngay sau khi phase hoàn thành, trước khi sang phase tiếp. Nếu đợi đến cuối để chạy hết một lần — debug sẽ khó hơn vì không biết phase nào introduce regression.

  5. 5

    Token budget: tính trước khi commit

    --deep --tdd cho một plan 9 phases có thể tốn 80–120k tokens. Nếu budget tight, consider --hard --tdd cho 3–4 phases critical nhất, --fast cho phases còn lại. Selective depth hơn là apply đồng đều.

§ 09

Pitfalls

Sáu pattern sai mình quan sát thấy — một số từ chính mình, một số từ codebase review. Mỗi cái có một dấu hiệu nhận biết rõ ràng.

P1
Quên --tdd ở /ck:cook
Plan được tạo với --tdd nhưng cook được gọi không có flag. Cook đọc phase file có 4 TDD sections nhưng không biết phải follow TDD flow — thực hiện theo default flow. Dấu hiệu: Tests Before section bị skip, refactor chạy trước khi regression coverage được viết.
P2
--deep cho task nhỏ
Dùng --deep cho task 1–2 file hoặc bug fix. Per-phase scout chạy nhưng không có gì đáng scout. Dấu hiệu: Phase file dày nhưng file inventory table chỉ có 1–2 dòng. Thời gian plan 4× so với thực thi.
P3
--tdd cho greenfield
Bật --tdd cho module viết mới hoàn toàn. Tests Before section trống hoặc cook tự phịa behavior để fill vào. Dấu hiệu: Tests Before chứa test cho hàm chưa tồn tại, assertion arbitrary không gắn với spec.
P4
Tin 100% phase file
Phase file từ --deep rất chi tiết — nhưng không phải oracle. Per-phase scout đọc code, không đọc được intent. Dấu hiệu: File inventory table có file listed nhưng thực tế không cần modify — scout đọc import graph nhưng không hiểu scope thật.
P5
--tdd chụp test cho code có bug sẵn
Tests Before "document behavior hiện tại" — kể cả khi behavior đó là bug. Refactor xong, Tests Before pass nhưng bug vẫn còn vì test assert behavior sai. Trước khi viết Tests Before: verify behavior hiện tại có đúng không, không phải chỉ assert output.
P6
--deep --tdd --parallel không ownership rõ
Combo mạnh nhất nhưng cũng nguy hiểm nhất khi file ownership không được define rõ trong phase file. Dấu hiệu: Hai parallel agents cùng modify một file — merge conflict hoặc tệ hơn là silent overwrite. Luôn verify ownership matrix trước khi chạy --parallel.
§ 10

Tóm lại

--deep và --tdd không phải silver bullets — chúng là tools có chi phí thật và trade-off thật. --deep đắt, nhưng với task đủ lớn và phức tạp, nó là khoản đầu tư: cook không phải đoán, phase file là bản đồ chi tiết, per-phase scout embed context trực tiếp vào nơi cần dùng.

--tdd giải quyết một lo âu rất cụ thể: "refactor này có phá behavior không?" Nó không làm code tốt hơn tự nhiên — nó tạo ra cấu trúc buộc bạn phải document behavior hiện tại trước khi thay đổi. Giá trị nằm ở kỷ luật đó.

Với port Rust → Go của mình, combo --deep --tdd là quyết định đúng nhìn lại. Không phải vì nó magic, mà vì project đúng với cả hai điều kiện: scope đủ lớn để --deep không overkill, và có behavior cụ thể cần bảo vệ để --tdd không trivial.

Ghi chú kỹ thuật

  • Per-phase scout là sub-feature của --deep, không available độc lập
  • --tdd có thể pass vào cả /ck:plan và /ck:cook — hai level independent
  • Token cost estimate: --fast ≈ 5k · --hard ≈ 20k · --deep ≈ 60–120k per plan
  • Regression Gate command được embed trong phase file — không cần nhớ, cook đọc trực tiếp
  • File inventory table format: action | path | rough_size | test_impact — reviewable trước khi execute

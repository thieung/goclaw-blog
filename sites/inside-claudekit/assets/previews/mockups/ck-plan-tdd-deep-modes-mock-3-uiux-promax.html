Khai phá sức mạnh của /ck:plan với --deep và --tdd

Mode mới --deep cho per-phase scouting và flag mới --tdd cho tests-first

22/04/2026 · 19 phút đọc ·
claudekit planning tdd refactor

Mình đã dùng /ck:plan hàng ngày được vài tháng rồi, nhưng gần đây mới bắt đầu hiểu sâu hơn về hai thứ mà bao lâu nay vẫn hay bỏ qua: flag --deep và flag --tdd. Trước đây mình cứ chạy plan với mặc định rồi thôi, thấy nó generate ra các file phase là xong.

Đến lúc dự án lớn lên, các phase bắt đầu có sự phụ thuộc chéo phức tạp, mình mới nhận ra mình đang dùng công cụ sai cách. --deep không chỉ là "plan kỹ hơn" — nó thay đổi hoàn toàn cách sub-agent tiếp cận từng phase. Còn --tdd thì buộc cả pipeline phải bắt đầu từ test trước, không phải implementation trước rồi viết test sau.

Bài này mình sẽ đi từ cơ chế bên trong, qua decision matrix thực tế, đến những cái bẫy mà mình đã vấp phải. Không phải tutorial từ đầu — giả định bạn đã dùng ClaudeKit một thời gian rồi.

01

Mode vs Flag

Khi mình lần đầu thấy --deep và --tdd trong docs, mình nghĩ cả hai đều là kiểu "bật thêm tính năng". Thực ra chúng hoạt động ở hai tầng khác nhau hoàn toàn: --deep là mode thay đổi cách plan được tạo ra, còn --tdd là flag thay đổi cách plan được thực thi. Nhầm lẫn hai cái này dẫn đến những kết hợp vô nghĩa hoặc kém hiệu quả.

Để phân biệt rõ hơn: mode ảnh hưởng đến output artifact (file phase, file research), còn flag ảnh hưởng đến pipeline behavior khi /ck:cook chạy sau đó. Một mode có thể dùng độc lập, một flag cần mode tương thích để phát huy tác dụng đúng nghĩa.

Khái niệm Tầng tác động Thay đổi gì
--deep (mode) Plan generation Per-phase scout, deeper context per phase file
--tdd (flag) Cook execution Tests-first template, regression gate ở cuối
bash 
# Kết hợp đúng: deep plan + tdd execution
/ck:plan --deep --tdd "refactor auth module"
02

--deep — "mode đại tu"

Mình gọi --deep là "mode đại tu" vì nó không chỉ generate thêm nội dung — nó spawn một sub-agent scout riêng cho mỗi phase. Thay vì một scout chạy toàn bộ codebase một lần rồi chia thông tin cho các phase, mỗi phase nhận được một lần scouting riêng với context window tập trung vào đúng phần code liên quan đến phase đó.

Pipeline của --deep

  1. Planner agent phân tích yêu cầu và tạo skeleton các phase
  2. Với mỗi phase, một scout sub-agent được spawn độc lập
  3. Scout đọc các file liên quan theo scope của phase đó
  4. Scout báo cáo về: dependencies, risk points, existing patterns
  5. Planner tổng hợp báo cáo, viết phase file với context cụ thể
  6. Các phase file được cross-check về dependency order
  7. Researcher agents chạy song song cho các topic kỹ thuật phức tạp
  8. Final plan.md được tạo với đủ link đến research và phase files

Per-phase scouting tốn nhiều token hơn nhưng đổi lại là các phase file chứa context thực sự hữu ích cho cook agent — thay vì generic instructions, cook agent nhận được thông tin về đúng function, đúng interface, đúng pattern cần thay đổi.

— Từ internal design notes của ClaudeKit
Chế độ Scout strategy Thích hợp cho
Default plan Single scout, toàn codebase Feature mới, greenfield
--deep Per-phase scout, focused scope Refactor, migration, codebase phức tạp
--deep --parallel Per-phase scout + parallel cook Large refactor với isolated phases
03

--tdd flag

Flag --tdd thay đổi template của phase file và thêm một bước regression gate vào cuối pipeline. Khi không có --tdd, phase file có cấu trúc: research → implementation → tests. Khi có --tdd, template đảo ngược: tests spec trước, rồi mới đến implementation để pass tests đó.

Điều quan trọng là --tdd không chỉ là thứ tự viết code — nó buộc cook agent phải định nghĩa behavior trước thông qua test cases, rồi mới implement. Đây là sự khác biệt về tư duy: thay vì "code xong rồi test xem có đúng không", cook agent phải "định nghĩa đúng là gì trước, rồi code để đạt được đó".

Phase Nội dung Sub-agent Gate
Tests Before Viết unit/integration tests từ spec Tester Compile pass
Refactor Implement để pass tests vừa viết Dev Tests green
Tests After Bổ sung edge case tests nếu cần Tester Coverage ≥ threshold
Regression Gate Chạy toàn bộ test suite CI runner No regressions
typescript 
// Ví dụ test-first với --tdd
// Bước 1: Viết test TRƯỚC khi có implementation
describe('AuthService.refreshToken', () => {
  it('should return new token pair when refresh token is valid', async () => {
    const result = await authService.refreshToken(validRefreshToken);
    expect(result).toHaveProperty('accessToken');
    expect(result.expiresIn).toBeLessThanOrEqual(3600);
  });

  it('should throw UnauthorizedException for expired refresh token', async () => {
    await expect(
      authService.refreshToken(expiredToken)
    ).rejects.toThrow(UnauthorizedException);
  });
});
// Bước 2: Implement AuthService.refreshToken để pass 2 tests trên
04

/ck:brainstorm trước khi plan

Trước khi chạy /ck:plan --deep, mình thường chạy /ck:brainstorm để làm rõ scope. --deep tốn token đáng kể, và nếu bạn chưa biết mình muốn refactor theo hướng nào thì per-phase scouting sẽ chỉ tạo ra nhiều context không nhất quán. Brainstorm giúp bạn converge về một approach trước rồi mới invest vào deep planning.

Bước Lệnh Mục tiêu
Khám phá /ck:brainstorm "vấn đề X" Hiểu không gian giải pháp
Lên kế hoạch /ck:plan --deep "approach Y" Deep plan với approach đã chọn
Thực thi /ck:cook --tdd <phase> Cook với TDD pipeline
05

Decision matrix

Tình huống Lệnh khuyến nghị
Feature mới, codebase sạch, spec rõ /ck:plan "feature X"
Feature mới, codebase phức tạp, nhiều dependencies /ck:plan --deep "feature X"
Refactor module đang có test coverage /ck:plan --tdd "refactor Y"
Refactor lớn, nhiều module liên quan /ck:plan --deep "refactor Y"
Migration (v1 → v2 API, DB schema) /ck:plan --deep "migrate Z"
Bug fix cần regression test rõ ràng /ck:plan --tdd "fix bug #123"
Task nhỏ, ít rủi ro, 1-2 file /ck:cook --fast (bỏ qua plan)
Refactor lớn cần TDD nghiêm ngặt /ck:plan --deep --tdd "refactor auth"
Migration phức tạp, parallel safe /ck:plan --deep --parallel "migrate to v3"
Legacy cleanup với coverage tăng dần /ck:brainstorm → /ck:plan --deep --tdd
06

Inside — cơ chế bên trong

6.1
Sub-agent counts

Số lượng sub-agent phụ thuộc vào số phase và mode. Với --deep, số scout agents bằng số phase. Bảng dưới tổng hợp cho plan có 5 phase điển hình.

Mode Scout Researcher Cook Tổng
Default 1 0–2 5 6–8
--deep 5 0–4 5 10–14
--tdd 1 0–2 5 + tester 7–10
--deep --tdd 5 0–4 5 + tester 11–15
--deep --tdd --parallel 5 0–4 5 parallel + tester 11–15 concurrent
6.2
--tdd là template-only ở plan stage

Điều mà docs không nói rõ: ở giai đoạn /ck:plan, --tdd chỉ thay đổi template của phase file — không có tester agent nào được spawn ở đây. Tester agent chỉ xuất hiện khi bạn chạy /ck:cook --tdd. Nghĩa là nếu bạn plan với --tdd nhưng cook không có --tdd, phase file có cấu trúc TDD nhưng cook agent sẽ ignore nó và implement theo flow thông thường.

6.3
Regression gate

Regression gate không phải là CI pipeline — nó là bước cuối của cook agent, chạy toàn bộ test suite của project và block merge nếu có regression. Khác với "Tests After" (chỉ kiểm tra tests của feature đang làm), regression gate kiểm tra toàn bộ. Đây là lý do --tdd nên dùng cho refactor chứ không phải greenfield: greenfield không có test suite sẵn để run regression.

6.4
Tóm tắt flow

Flow đầy đủ khi dùng --deep --tdd: planner → N scouts (song song) → researcher (song song với scouts) → planner tổng hợp → phase files với TDD template → cook nhận phase → tester viết tests → dev implement → tester bổ sung → regression gate. Toàn bộ flow này transparent trong plans/ directory — bạn có thể trace từng step.

07

Cách chạy và mẫu prompt

Khi viết task description cho /ck:plan --deep, cần cung cấp đủ 4 yếu tố sau:

  • Phạm vi — file/module nào nằm trong scope của plan này
  • Ràng buộc — backward compatibility, không được break API nào
  • Tooling — test framework, build tool, deployment target
  • Kết quả mong đợi — metric cụ thể: coverage %, performance budget, migration %
bash · ví dụ 1 — refactor với tdd 
/ck:plan --deep --tdd "Refactor auth module
Scope: src/auth/**, không đụng đến src/user/**
Constraints: giữ nguyên public API của AuthService
Tooling: Jest + Supertest, NestJS
Expected: coverage >= 85%, tất cả existing tests vẫn pass"
bash · ví dụ 2 — migration 
/ck:plan --deep "Migrate từ Prisma v4 sang v5
Scope: src/**, database/schema/**
Constraints: không thay đổi data model, zero downtime migration
Tooling: Prisma CLI, PostgreSQL 15
Expected: 100% feature parity, migration script tested on staging"
bash · ví dụ 3 — parallel với ownership rõ 
/ck:plan --deep --parallel "Split monolith thành services
Phase 1 owns: src/auth/** → service-auth/
Phase 2 owns: src/billing/** → service-billing/
Phase 3 owns: src/notification/** → service-notification/
Shared: packages/core/** (read-only for phases 1-3)"
08

Best practices

1
Luôn đọc phase file trước khi cook
Phase file từ --deep có context cụ thể mà planner đã research. Đọc kỹ trước khi chạy /ck:cook giúp bạn catch được các assumption sai hoặc scope creep trước khi mất thời gian cook. Đặc biệt kiểm tra phần "Key Insights" và "Risk Assessment".
2
Dùng --tdd chỉ khi có test infrastructure sẵn
--tdd giả định bạn có test runner, test utilities, và một test suite hiện có để chạy regression. Nếu project chưa có test setup thì nên dùng /ck:cook thông thường trước để setup infrastructure, rồi mới bật --tdd cho các phase sau.
3
Giữ ownership rõ ràng khi dùng --parallel
Khi kết hợp --deep --tdd --parallel, mỗi phase PHẢI có file ownership rõ ràng trong task description. Không có ownership rõ thì parallel agents sẽ edit cùng file và tạo conflict. Viết glob patterns cụ thể vào prompt, không để planner tự suy luận.
4
Review research reports trước khi approve plan
Trong --deep mode, researcher agents tạo reports trong plans/{slug}/research/. Đọc các reports này trước khi approve plan vì đôi khi researcher phát hiện constraint mà planner không incorporate vào phase files. Bạn có thể edit phase file trực tiếp trước khi cook.
5
Kết hợp /ck:brainstorm khi scope không rõ
--deep tốn nhiều token. Đầu tư token vào brainstorm trước để làm rõ approach, sau đó plan với scope đã converge sẽ tiết kiệm hơn nhiều. Pattern: /ck:brainstorm (làm rõ "làm gì") → /ck:plan --deep (làm rõ "làm thế nào") → /ck:cook (thực thi).
09

Pitfalls

Sáu bẫy phổ biến mà mình — và những người mình biết dùng ClaudeKit — đã vấp phải. Không phải lỗi của tool, mà là lỗi của mental model khi dùng tool.

P1
Quên --tdd ở /ck:cook
Plan với --tdd tạo phase file có TDD template, nhưng nếu bạn cook mà không truyền --tdd, cook agent sẽ implement theo thứ tự thông thường. Phase file có structure TDD nhưng không ai enforce nó. Luôn nhất quán: plan --tdd thì cook cũng phải --tdd.
P2
--deep cho task nhỏ
Dùng --deep cho task 1–2 file là lãng phí token. Per-phase scouting sinh lợi khi scope rộng, có nhiều dependencies, và phase files cần context riêng biệt nhau. Với task nhỏ, default plan + một lần scout là đủ.
P3
--tdd cho greenfield
Greenfield không có test suite để run regression — regression gate sẽ pass trivially (không có gì để break). Hơn nữa, viết tests trước khi có bất kỳ interface nào buộc tester agent phải đoán API design. Dùng --tdd khi refactor, không phải khi build từ đầu.
P4
Tin 100% phase file
Dù --deep cho context tốt hơn, phase file vẫn là output của AI agent — có thể có assumption sai về codebase. Luôn review phase file trước khi cook. Chỉnh sửa trực tiếp nếu cần. Phase file là living document, không phải contract cứng.
P5
--tdd chụp test cho code có bug sẵn
Nếu bạn dùng --tdd để refactor code đang có bug, tester agent có thể viết tests mirror behavior có bug của code hiện tại. Tests pass, nhưng behavior vẫn sai. Trước khi refactor với TDD, hãy fix bug trước — hoặc viết failing test cho bug đó như một phần của TDD spec.
P6
--deep --tdd --parallel không ownership rõ
Kết hợp cả ba flags mà không define file ownership rõ ràng là công thức để conflict. Parallel agents viết tests cho cùng shared module, rồi cook agents implement khác nhau cho cùng file. Kết quả là merge conflict hoặc tệ hơn: silently broken behavior.
10

Tóm lại

--deep và --tdd không phải "tính năng thêm vào cho chắc" — chúng là tool cho những tình huống cụ thể. --deep khi scope rộng và context per-phase matter, --tdd khi bạn đang refactor existing code và cần regression safety. Dùng đúng tình huống thì chúng thực sự mạnh; dùng sai thì chỉ tốn thêm token mà không thêm giá trị.

Mental model mình hay dùng: --deep là "đầu tư vào sự hiểu biết trước khi làm", còn --tdd là "đầu tư vào sự an toàn trong khi làm". Nếu bạn confident về codebase (đã làm nhiều), không cần --deep. Nếu bạn làm feature mới (không có regression risk), không cần --tdd.

Cuối cùng, ClaudeKit chỉ là tool. Chất lượng của plan phụ thuộc nhiều vào chất lượng của task description bạn viết. --deep giúp agent hiểu codebase tốt hơn, nhưng nếu bạn mô tả sai problem thì agent vẫn sẽ plan sai — chỉ là sẽ plan sai một cách kỹ lưỡng hơn mà thôi.

Ghi chú kỹ thuật
  • Per-phase scouting dùng context window riêng — không shared giữa các scout agents
  • Researcher agents trong --deep mode bị cap ở max 4 agents chạy song song
  • Regression gate chạy pnpm test (hoặc npm test) trong project root — cần test script sẵn
  • --tdd template require: test framework config, test utilities, ít nhất 1 existing test file
  • File ownership trong --parallel dùng glob patterns theo gitignore syntax

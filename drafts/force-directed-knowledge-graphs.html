Knowledge Graph (KG) duoc gioi thieu trong GoClaw tu v2.15.0, cho phep AI agent xay dung bo nho ngu nghia lau dai tu hoi thoai voi nguoi dung. Thay vi chi luu cac doan chat roi le, agent hieu duoc moi quan he giua cac thuc the — ai dang lam du an nao, deadline nao sap den, cong ty nao lien ket voi nhau.

Knowledge Graph (KG) was introduced in GoClaw starting v2.15.0, enabling AI agents to build long-term semantic memory from user conversations. Instead of storing isolated chat fragments, agents understand relationships between entities — who works on which project, upcoming deadlines, and organizational connections.

知识图谱（KG）从 v2.15.0 开始引入 GoClaw，让 AI 智能体能够从对话中构建长期语义记忆。不再只是存储孤立片段，智能体能理解实体间的关系——谁在做哪个项目、即将到来的截止日期以及组织关联。

ナレッジグラフ（KG）は v2.15.0 から GoClaw に導入され、AIエージェントが会話から長期的なセマンティックメモリを構築できます。孤立した断片ではなく、エンティティ間の関係を理解します。

v2.47.0 tap trung hai mang lon: React force-directed visualization hien thi toan bo mang luoi tri thuc cua agent, va 7 toi uu hieu nang dam bao graph 50 node chay muot ma khong bi jank. Xem tai lieu day du tai docs.goclaw.sh/knowledge-graph.

1. Embedding Provider & Model

KG can embedding model de tao vector cho semantic search va deduplication. Vao Dashboard → Settings → Embedding, chon provider (OpenAI, Anthropic…) va model — vi du text-embedding-3-small. Neu khong cau hinh, KG van hoat dong nhung chi dung Full-Text Search, khong co kha nang tim kiem ngu nghia.

KG needs an embedding model to generate vectors for semantic search and deduplication. Go to Dashboard → Settings → Embedding, select provider (OpenAI, Anthropic…) and model — e.g. text-embedding-3-small. Without this, KG still works but falls back to Full-Text Search only.

KG 需要嵌入模型为语义搜索和去重生成向量。前往 Dashboard → Settings → Embedding，选择提供商（OpenAI、Anthropic 等）和模型——如 text-embedding-3-small。未配置则仅使用全文搜索，无语义搜索能力。

KGはセマンティック検索と重複排除のベクター生成に埋め込みモデルが必要。Dashboard → Settings → Embedding でプロバイダーとモデル（例：text-embedding-3-small）を選択。設定しない場合は全文検索のみ。

2. Extraction Provider & Model

Extraction dung LLM de nhan dien entities va relations tu van ban hoi thoai. Vao Dashboard → Settings → Knowledge Graph, chon extraction provider va model (GPT-4o, Claude 3.5 Sonnet…). Model extraction anh huong truc tiep den chat luong du lieu graph.

Extraction uses LLM to identify entities and relations from conversation text. Go to Dashboard → Settings → Knowledge Graph, select extraction provider and model (GPT-4o, Claude 3.5 Sonnet…). The extraction model directly affects graph data quality.

提取使用 LLM 从对话文本中识别实体和关系。前往 Dashboard → Settings → Knowledge Graph，选择提取提供商和模型（GPT-4o、Claude 3.5 Sonnet 等）。提取模型直接影响图谱数据质量。

抽出はLLMを使用して会話テキストからエンティティと関係を識別。Dashboard → Settings → Knowledge Graph で抽出プロバイダーとモデル（GPT-4o、Claude 3.5 Sonnet など）を選択。抽出モデルはグラフデータ品質に直接影響。

Entity Mass / 实体质量 / エンティティ質量

Mass — Kich thuoc va luc day

Entity Mass — Size & Repulsion

实体质量——尺寸与排斥力

エンティティ質量——サイズと反発力

Moi entity type co gia tri mass quyet dinh kich thuoc node va cuong do luc day trong simulation. Mass cao hon = day manh hon (charge = -300 × mass) va chiem vung khong gian lon hon (collision = 55 + mass × 5), tu nhien tro thanh trung tam graph.

Each entity type has a mass value determining node size and repulsion in simulation. Higher mass = stronger repulsion (charge = -300 × mass) and larger space reservation (collision = 55 + mass × 5), naturally becoming graph centers.

每种实体类型都有质量值，决定节点大小和模拟中的排斥强度。质量越高 = 排斥力越强（charge = -300 × mass），空间占用越大（collision = 55 + mass × 5），自然成为图的中心。

各エンティティタイプには質量値があり、シミュレーションのノードサイズと反発強度を決定。質量が高いほど反発が強く（charge = -300 × mass）、空間占有が大きく（collision = 55 + mass × 5）、自然にグラフの中心になります。

Hub Detection — Nhan dien node trung tam

Hub Detection

枢纽检测

ハブ検出

Node duoc danh dau la hub khi co degree >= 4 (ket noi voi it nhat 4 node khac). Hub nodes hien thi voi glow effect va scale lon hon, giup nguoi dung nhan ra cac thuc the trong tam trong network ngay lap tuc.

A node is marked as a hub when its degree >= 4 (connected to at least 4 other nodes). Hub nodes render with a glow effect and larger scale, letting users instantly identify central entities.

当节点的度数 >= 4（与至少4个其他节点相连）时，该节点被标记为枢纽。枢纽节点以发光效果和更大比例渲染，让用户立即识别核心实体。

ノードの次数 >= 4（少なくとも4つの他のノードに接続）のときハブとしてマーク。ハブノードはグロー効果と大きなスケールでレンダリングされ、中心的なエンティティを即座に識別できます。

7 Performance Optimizations in v2.47.0

Tap trung vao rendering path — giam allocation, giam re-render, va align fetch limit voi render limit. Ket qua: graph 50 nodes render muot voi zero jank khi pan/zoom.

Focused on the rendering path — reducing allocation, reducing re-renders, aligning fetch limit with render limit. Result: 50-node graph renders smoothly with zero jank on pan/zoom.
